Upstream `verygoodplugins/whatsapp-mcp` PR #112 (env-driven stdio/http/sse
transport) defaults `WHATSAPP_MCP_PORT` to **8000** — both in `resolve_port()`
and the `FastMCP(...)` constructor — and its `tests/test_transport.py` asserts
8000. SPEC-WAMCP-001 §4.1.1 mandates a default of **8080**.

A verbatim cherry-pick therefore ships a contract-violating default and would
pass its own (wrong) tests, so CI would NOT catch it. When implementing
P-IMPL-1, reconcile the default to 8080 in four places in one pass — code
(`resolve_port` + `FastMCP`), `whatsapp-mcp-server/.env.example`, `README.md`,
and the PR-112 test assertions — then `grep -rn '8000'` to confirm none remain.
This is the CLAUDE.md "change a default → grep the whole surface in one commit"
doctrine applied to a cherry-pick. Note the MCP client token
`WHATSAPP_MCP_TOKEN` (added by this mission's auth phase) is a different layer
from the existing bridge token `WHATSAPP_BRIDGE_TOKEN` — do not conflate them.
